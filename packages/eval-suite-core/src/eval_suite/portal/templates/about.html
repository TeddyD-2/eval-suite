about this portal

CONTRACT_VERSION = {{ contract_version }}

The contract version is the semver-versioned API surface that third-party plugins target. Required Policy / Task / Adapter / Manifest methods are the stability surface; optional hooks (instruction_for, extract_image, canonical_axis_map, can_drive, ACTION_SPACE_HINT) are not.

Source: eval_suite/contracts.py in the GitHub repo.

what this portal proves

  • Manifest integrity at submission. The signed manifest's content hash + Ed25519 signature mean the submitter held the key and the inputs they declared are bit-for-bit consistent with the recorded numbers.
  • Submitter identity at submission. When the portal enforces an allow-list, the submitter's claimed identity is overwritten with the allow-list canonical identity — spoofing an identity string the submitter doesn't have a key for fails closed.
  • Cross-device corroboration. Two distinct submitters submitting the same run_id is recorded as independent corroboration (see the corroborators section on each submission detail page).

what this portal does NOT prove

The ledger at /ledger is append-only by convention only — the portal code only appends, never rewrites. It is not cryptographically tamper-evident. The portal operator could selectively rewrite or hide entries. v0 proves the submitter held the key (Ed25519 over the manifest is tamper-evident at submission time) but does not prove the ledger of accepted submissions has not been edited. Sigstore migration in v1 closes the second hole.

what the compare page is asserting

Compare renders the canonical 4-dimension profile (language / visuals / physics / embodiment) for both runs, computed from each manifest's cell payload. It does not render raw per-cell axes. That's the point: runs from different task families (Google Robot vs WidowX vs Go1) have different raw axes; the canonical-dim layer is what makes them comparable, and is the project's central claim.

Worst-canonical-dim is the headline ranking metric: a model is judged by its weakest dimension, not its average, because in deployment the weakest dimension is what fails.

calibration tiers

Every (model, task) result carries an A / B / C calibration tier:

  • Tier A — paired real-robot data exists; sim and real are compared via Pearson r with confidence intervals.
  • Tier B — published real-robot numbers exist for at least one cell; sim and real are reported side-by-side with absolute delta. No correlation claim.
  • Tier C — no real-robot data; sim score reported alone with a "no calibration" flag.

Full definitions and the C→B→A→A+ upgrade path: see EXTENSION.md §7.
